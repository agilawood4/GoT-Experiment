## **Generic 方法：统一两子问题 GoT**

`graphwiz_got.py` 里的 Generic 方法是一个完全任务无关的 GoT 模版，其固定流程是：

```python
原问题
 → 让模型把问题拆成恰好两个子问题，并且必须返回 JSON
 → 每个子问题分别生成 3 个候选解
 → 用 graphwiz_format_score 做格式/输出质量打分
 → 每个子问题保留 1 个最佳答案
 → 聚合两个子问题答案
 → final score
 → KeepBestN(1)
 → GroundTruth
```

提示词统一设计为：

 `DECOMPOSE_PROMPT` 负责拆分，`SOLVE_SUBPROBLEM_PROMPT` 负责解子问题，`AGGREGATE_PROMPT` 负责回到原题。parser 在 phase0 时会先解析 JSON；如果模型没返回合法 JSON，就调用 `utils.default_decomposition()` 做兜底拆分

## **九大类任务差异化 GoO 设计**

### **Connectivity**

分支有两个：

*  `PathWitness`：只负责找一条从 source 到 target 的显式路径
*  `SeparationCheck`：只负责找明确的断开证据，比如不同连通分量、不可达扩展、孤立节点

聚合规则：

* 只要 `PathWitness` 找到任何显式路径，就必须答 Yes；
* 只有在**没有路径证据**且 `SeparationCheck` 给出**具体断开证据**时，才能答 No；
* 没有发现断开证据本身不构成 No 的依据，本质上是 **constructive witness 优先于 negative speculation** 的原则

验证方面：

* `search_score` 会惩罚 PathWitness 没有序列、或者错误输出 No；也会惩罚 SeparationCheck 里出现路径内容
* `ground_truth` 用 `utils.graph_connectivity_truth()` 做程序化判定

整体上是一个显式可达证据优先，断开结论必须有**硬证据**的 yes/no 证据对抗框架

### **Bipartite**

这个任务也是 yes/no，但证据形式换成了**构造合法划分**和**构造反例奇环**

两个分支：

* `TwoColorAttempt`：尝试构造 two-coloring / two-partition
* `OddCycleSearch`：尝试找到奇环见证

聚合规则：

* 若存在有效二染色解释且没有具体奇环证据，则 Yes
* 若存在具体奇环证据，则 No
* 本质上是“正向可行性证明 vs 反向反例证明”

验证方面：

* `search_score` 会检查回答里是否真的包含 color/partition 或 odd cycle 等关键词
* `ground_truth` 用 `utils.graph_bipartite_truth()`

把二分图判定转成找合法二染色和找奇环反例的双证据保险框架

### **Cycle**

cycle 跟 connectivity 很像，但正解的对立面换成了 acyclic evidence

两个分支：

* `CycleWitness`：找一条显式环；
* `AcyclicCheck`：只找具体无环证据，比如 elimination、无 back-edge、DAG 型解释

聚合规则：

### **Topology**

这是九类问题中**最难**的一类

* 小中大图自适应 prompt
* 自定义 `TopologyParser`，在 parser 阶段把模型给出的候选序列当成偏好顺序，再用 **Kahn 算法程序化修复**，保证最终输出合法

分支只有两个：

* `ConstraintExtraction`：抽少量 anchor precedence constraints
* `CandidateOrdering`：产出一个完整候选拓扑序

而且 prompt 按图规模分 bucket：

* small / medium / large
* 大图时显式限制模型不要重述边集、不要写长解释、尽量只给最终序列；小图允许少量 reasoning

### **Shortest Path**

标准的**数值型三分支框架**。

三个分支：

* `CandidatePathComputation`：构造一个强候选最短路并精确算总权重；
* `DistanceRelaxation`：用 Dijkstra/Bellman-Ford 风格的松弛思想推距离；
* `AlternativeAudit`：把候选和其他 plausible alternatives 比较，排除更长、成环、无依据路径。 

聚合规则：

* 最终答案输出的是**最短路权重**而不是路径串
* 优先采用同时被“候选路径”和“松弛推理”支持的值
* 排除循环或明显更长的备选

正解验证：

* 先解析边和 source-target
* 若有负边，用 Bellman-Ford
* 否则用 Dijkstra
* 不可达则允许 “No path”

我们设计的 Shortest path 的 GoO 是一个候选值 + 一个全局距离传播视角 + 一个反例审计视角三方交叉验证

### **Flow**

三个分支：

* `SourceSinkBounds`：算上界，来自 source 出边、sink 入边、cut、bottleneck
* `AugmentingPathPlan`：构造可行下界，强调 augmenting paths 组合时不能共享边超容量
* `OptimalityCheck`：证明当前最好可行流已经等于最强有效上界，从而认证最优

聚合规则：

* 最终只输出 maximum flow 数值
* 仅信任既是可行流、又没被更紧上界否定的数值
* 拒绝因为不兼容路径叠加导致的 over-counting，本质上相当于把 max-flow 的核心推理拆成了**上界—下界—夹逼证最优**

Flow 的 GoO 是一个非常标准的上界/下界/最优性三段式框架

### **Triangle**

这个任务是数值型，但不是最短路那种全局传播，而是**组合搜索 + 局部验证**

三个分支：

* `HighWeightCandidateScan`：先看高权重点，筛出可能的 triangle 候选
* `TriangleVerification`：检查三点是否真的是 3-clique，并精确算点权和
* `AlternativeComparison`：把已验证最好三角形和其他 plausible alternatives 比较，保留最大和

聚合规则：

* 必须是三条边都存在的真 3 角形
* 目标值是**节点权重和**，不是边权

程序 GT 检验：穷举所有三元组，检查三条 pairwise edges 是否都在，再取最大点权和

### **Hamilton**

三个分支：

* `DegreeAndEndpointHeuristics`：分析度为 0/1 的点、强制端点、断裂部分、方向约束
* `CandidateCoverWalk`：尝试构造一次覆盖所有节点且每点恰好一次的 walk
* `ObstructionCheck`：找明确障碍，比如孤立点、端点过多、不可达、死路

聚合规则：

* 只有当完整 cover-walk 有可信支撑时才偏向 Yes
* 只有当存在具体 obstruction 时才偏向 No
* 最终不输出长路径，只输出 Yes/No

GroundTruth 也很强：不是只看 gold，而是对 n≤22 的图直接做 bitmask DFS 精确判定是否存在 Hamilton path；图太大再退回 gold/fallback

### **Substructure**

这是一个很典型的**图匹配型三分支框架**

三个分支：

* `PatternRoleExtraction`：先抽取 pattern graph 里的角色签名，比如 in-degree/out-degree、anchor、稀有 motif
* `CandidateMapping`：尝试提出一个 injective mapping
* `EdgeCoverageAudit`：检查该映射下每条 pattern edge 是否都被 host graph 保留，或者指出缺失边

聚合规则：

* 只有具体 injective mapping + 所有 required pattern edges 都保留时才能 Yes
* 只要某个关键 required edge 缺失，就支持 No。

Substructure 的 GoO 是先缩小匹配空间，再提映射，再逐边审核的匹配审计框架

```python
Substructure 的 GoO
```
